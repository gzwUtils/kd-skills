# 文档管理（Docs-as-Code）

目标：让文档 **可发现、可维护、可演进**，并融入研发流程（而不是“写一次就过期”）。

## 适用范围
- Java/Spring Boot 项目（单体/微服务）
- Git 仓库内维护文档（Markdown/Asciidoc）

## 文档体系（建议最小集）
- **README**：项目概述、启动方式、关键链接、常见命令
- **架构文档**：上下文/组件/边界/数据流/关键决策（建议用 ADR）
- **API 文档**：OpenAPI + 示例（成功/失败/边界）
- **开发指南**：本地环境、调试、配置、常见问题
- **部署与运维**：发布/回滚、配置项、监控告警、容量规划
- **故障排查**：问题分层定位路径、常见故障手册、RCA 模板

## 工作流（建议）
1. **文档与代码同 PR**：重要变更必须带文档更新（或说明“不需要”的原因）。
2. **自动化生成**：API 文档由 Springdoc/OpenAPI 自动生成；变更在 CI 做校验。
3. **变更可追溯**：关键架构决策使用 ADR（Architecture Decision Record）。
4. **定期体检**：按月/季度对“最常用文档”做可用性检查（过期/缺失/链接失效）。

## 工具建议
- API：Springdoc OpenAPI + Swagger UI
- 代码文档：Javadoc（面向公共 API/复杂逻辑）
- 架构：C4/ADR（Markdown）
- 校验：markdownlint/link checker（可选）

## 检查清单
- [ ] README 可一键启动（含必要命令/配置）
- [ ] 架构边界与关键决策可追溯（ADR）
- [ ] OpenAPI 可访问且与实现一致（CI 校验）
- [ ] 变更影响/回滚/兼容策略在文档中可找到
- [ ] 常见故障有定位路径与处理手册
