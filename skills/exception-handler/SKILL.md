# 异常处理器（统一错误模型）

目标：在 Spring Boot 3.x 中实现 **一致的错误响应**、**可观测的异常日志**、**可治理的错误码**，避免把内部异常细节泄露给客户端。

## 适用范围
- Spring MVC/WebFlux
- 面向外部/内部 API 的统一错误响应

## 设计要点

### 异常分类（建议）
- **业务异常**：可预期、可提示（如库存不足、状态不允许）
- **校验异常**：参数校验失败（字段级错误可返回）
- **权限异常**：未认证/无权限
- **资源异常**：Not Found、冲突（版本/并发）
- **系统异常**：不可预期（DB/MQ/第三方依赖故障）

### 统一错误响应（建议字段）
- `code`：稳定的业务错误码（用于前端/调用方逻辑与告警聚合）
- `message`：面向用户/调用方的可读信息（避免暴露内部细节）
- `traceId/requestId`：用于链路追踪
- `details`：字段校验错误/上下文（可选，需脱敏）
- `timestamp/path`：辅助定位（可选）

> Spring 6 支持 `ProblemDetail`，也可自定义 ErrorResponse；关键是“统一 + 稳定 + 可追踪”。

## 工作流（建议）
1. 定义错误码规范（前缀/模块/编号）与枚举/常量落地方式。
2. 定义业务异常基类：携带 `code` 与可选 `args/details`。
3. 用 `@ControllerAdvice` 映射异常 → HTTP 状态码 + 统一错误响应。
4. 日志规范：系统异常打 error + stacktrace；业务异常打 warn/info + 上下文。
5. 对外隐藏内部信息：不要直接返回 Exception message/stacktrace。

## 检查清单
- [ ] 错误码体系稳定（可检索、可治理、可告警聚合）
- [ ] `@ControllerAdvice` 覆盖校验/业务/权限/系统异常
- [ ] 日志包含 traceId + 关键业务字段（并脱敏）
- [ ] 不泄露内部异常细节（SQL、堆栈、密钥等）
- [ ] 调用方可根据 code 做稳定处理
