# 性能优化器（Measure → Optimize → Verify）

核心原则：**先度量、再定位、后优化、最后验证**。避免“凭感觉调参”。

## 适用范围
- Java 17 + Spring Boot 3.x + MySQL/Redis/MQ 等常见栈

## 工作流（推荐）
1. **定义指标**：延迟/吞吐/错误率/资源占用；明确目标（SLO）。
2. **定位瓶颈**：先看观测（指标/日志/trace），再做 profiling（CPU/内存/锁/IO）。
3. **提出假设**：瓶颈在 DB/缓存/外部依赖/线程池/算法/序列化？
4. **最小改动优化**：一次只改一个变量，避免混合优化导致无法归因。
5. **回归与压测**：验证效果（平均/TP99）与副作用（内存/GC/错误率）。

## 高频优化点

### 数据库
- 索引与 SQL：避免全表扫描、隐式转换、深分页；关键 SQL 必须 `EXPLAIN`。
- 连接池：合理配置最大连接/超时，避免连接耗尽。
- 批处理：批量写入/更新，避免 N+1 与循环写库。

### 缓存（Redis/本地）
- 读多写少优先缓存；明确 TTL 与失效策略。
- 防穿透/击穿/雪崩：布隆过滤/互斥锁/热点预热/随机过期。
- 避免大 key、热 key；关注序列化开销与网络开销。

### 应用与 JVM
- 算法与数据结构：减少 O(n^2)；避免不必要的复制与装箱。
- 线程池：隔离不同业务/依赖；设置队列与拒绝策略；避免共享线程池拖垮。
- GC：不要盲目调参；先看分配率/停顿/堆外内存/对象生命周期。

## 检查清单
- [ ] 有明确的性能目标与基线数据
- [ ] 关键链路可观测（指标/日志/trace）
- [ ] 关键 SQL/缓存命中率有验证
- [ ] 优化可归因（一次改动一个变量）
- [ ] 压测/回归覆盖关键路径与失败路径
