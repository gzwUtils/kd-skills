# 架构审查（Architecture Reviewer）

## 适用范围
- Java 17 / Spring Boot 3.x（单体或微服务均可）

## 工作流（建议）
1. **澄清目标与约束**：要解决什么问题？SLA/一致性/成本/交付周期约束是什么？
2. **识别边界与依赖**：模块/领域/服务边界是否清晰？依赖方向是否单向可控？
3. **验证关键链路**：事务边界、并发、幂等、重试、超时、降级、回滚策略是否明确？
4. **评估演进成本**：配置、部署、观测、扩容、兼容性、数据迁移、技术债影响。
5. **给出可执行建议**：必须调整/建议优化/风险点/验证清单。

## 审查要点（Must）
- **分层职责**：Controller/Service/Domain/Repository 责任边界清晰，避免“到处写业务逻辑”。
- **依赖方向**：核心业务不依赖基础设施；尽量通过接口/适配器隔离外部系统。
- **一致性策略**：跨聚合/跨服务一致性（强一致/最终一致/TCC/消息事务）要有设计依据。
- **幂等与去重**：写接口/消息消费端必须明确幂等键、去重策略与并发竞争处理。
- **失败路径**：超时、重试、熔断、降级、补偿、死信与告警策略清晰。
- **兼容与演进**：API/事件/DB 的向前兼容策略明确（版本、字段、语义）。

## 常见反模式（Anti-patterns）
- 业务逻辑堆在 Controller/Mapper/Listener，导致测试困难、复用差。
- “贫血模型”：所有对象都是 DTO，核心规则散落在各处 if-else。
- 事务跨远程调用/事务范围过大，导致锁等待和级联失败。
- 过早/过度拆分微服务，引入分布式复杂度但没有收益。
- 模块边界不清导致循环依赖、随意跨模块调用。

## 验证清单（可用于落地）
- [ ] 关键接口有契约：DTO 校验、错误码、幂等约束、权限约束。
- [ ] 关键链路有回归：单测/集成测/契约测试或最小回归脚本。
- [ ] 关键失败路径可观测：结构化日志 + 指标 + Trace。
- [ ] 新增依赖有治理：版本统一、License/安全评估、替换策略。
