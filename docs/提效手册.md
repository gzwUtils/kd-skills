# 提效手册（Java 架构师 & AI 程序员）

## 目标
- 把“规范”从口头共识变成 **可执行资产**（Skill / 清单 / 自动化门禁）
- 把重复劳动交给工具（生成/检查/文档/回归），人专注在业务与架构决策
- 用 AI 做“第二双眼睛”：更快发现风险点，但不替代工程验证

## 从 0 到 1 的落地路径（建议 2～4 周）
1. **第 1 周：建立最小门禁**
   - 统一提交信息（Conventional Commits）与 PR 模板
   - 接入最小静态检查（SpotBugs/Checkstyle/Formatter 任选其一）
2. **第 2 周：把 Skill 嵌进日常**
   - PR 必须回答：目的/影响面/验证方式/回滚方案
   - 用 `code-reviewer` + `security-expert` 做 PR Review 清单
3. **第 3 周：引入可观测与压测基线**
   - 关键链路接入日志字段、指标与 trace（`observability-engineer`）
   - 建立 TP95/TP99 基线与回归门槛
4. **第 4 周：工程化沉淀**
   - 把高频问题固化为：脚手架/模板/生成器/IDE 检查项
   - 为架构关键决策引入 ADR（`documentation` + `architecture-reviewer`）

## “Skill → 自动化门禁”的拆分思路
- **能确定性检查的**：交给静态工具（Checkstyle/SpotBugs/PMD/Sonar/Dependency-Check）
- **需要上下文判断的**：交给 AI（架构权衡、边界设计、变更影响评估）
- **必须可验证的**：交给测试/压测/回归脚本（JUnit/Testcontainers/Gatling 等）

## IDEA 提交前检查（Commit-time Agent）
仓库已提供 IntelliJ 插件骨架：`integrations/intellij-plugin/`。

建议策略：
- 提交前只做 **轻量** 检查：AI Review + 快速静态检查（避免阻塞太久）
- 真正的强门禁放在 CI：单测/集成测/扫描/构建制品

使用要点：
- `Skills root path` 指向本仓库根目录（或你发布到 GitHub 后 clone 的目录）
- `Skill names` 为空=全量；建议默认：`code-reviewer,security-expert,exception-handler,performance-optimizer,unit-tester`
- 使用 OpenAI-Compatible 网关时，优先走“企业代理/内网网关”，并开启脱敏与审计

## PR/评审提效模板（建议复制到 PR 描述里）
1. **目的**：解决什么问题？为什么现在做？
2. **范围**：改了哪些模块/接口/表/配置？
3. **风险点**：一致性/并发/权限/性能/回滚风险有哪些？
4. **验证**：如何验证？（用例、日志、压测、截图、回归结果）
5. **回滚**：如何回滚？（代码/配置/数据迁移）

## 代码生成提效（减少重复 CRUD）
建议把 `code-generator` 从“提示”升级为“资产”：
- 固化模板：Controller/Service/DTO/Mapper/错误码/日志
- 默认接入校验与异常处理：生成即合规
- 生成即带测试：至少生成 Service 单测骨架

## 数据库与性能提效（避免上线后救火）
- 设计阶段用 `database-designer` 做表与 SQL 评审清单（EXPLAIN、索引、分页）
- 上线前做性能基线：对关键接口做最小压测（TP95/TP99、错误率、资源占用）
- 缓存治理：穿透/击穿/雪崩/热 key 预案（`performance-optimizer`）

## 安全提效（把安全左移）
- PR 门禁：新增接口必须说明鉴权/权限点与输入校验
- 依赖治理：CVE 扫描 + SBOM（`ci-cd-engineer`）
- 日志脱敏与审计：默认不打印敏感信息（`security-expert` + `observability-engineer`）

## 文档提效（Docs-as-Code）
- ADR：每个“不可逆/高影响”决策写一条 ADR（5～15 分钟即可）
- OpenAPI 自动化：接口变更自动更新文档，CI 校验一致性
- 故障手册：把复盘结论沉淀为“下一次能用的操作手册”

## Skill 仓库治理（发布到 GitHub 的建议）
- 版本化：用 tag（`v0.1.0`）标记一次规范升级
- 评审机制：Skill 变更也走 PR + Review（避免规范漂移）
- 结构约束：每个 Skill 目录只放 `SKILL.md`（和必要的 references/scripts）
- 反馈闭环：每月统计“最常见问题 TOP10”，把它们固化为门禁或模板

## 风险与边界（必须明确）
- **隐私/合规**：提交差异发给 LLM 前，明确数据分类与脱敏策略
- **AI 仅做建议**：最终以测试/观测/压测结果为准
- **不做大重构**：提效优先做“可渐进”的工程化改造（模板/门禁/脚手架）
